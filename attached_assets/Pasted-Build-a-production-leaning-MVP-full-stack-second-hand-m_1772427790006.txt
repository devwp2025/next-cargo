8) MOCK PAYMENT GATEWAY (Payment model A)
========================================================
Flow:
- On "Buy Now":
  1) Create Order status = pending_payment
  2) Reserve product to prevent double selling
  3) Create Payment(provider=mockpay, status=requires_action, sessionId unique)
  4) Redirect to /pay/mock/[sessionId]

Mock checkout page: "/pay/mock/[sessionId]"
- Show order summary
- Buttons: Pay Success / Pay Fail / Cancel

Webhook endpoint:
POST /api/webhooks/mockpay
- Verify header: x-mockpay-signature === MOCKPAY_WEBHOOK_SECRET
- Events:
  - payment.succeeded:
      Payment.status = succeeded
      Order.status = paid
  - payment.failed OR payment.canceled:
      Payment.status = failed
      Order.status = canceled
      Product.status = active (release reservation)

IMPORTANT:
- NEVER set Order paid from the client.
- Only webhook updates payment/order final status.

Idempotency:
- Webhook handling must be idempotent:
  - If a payment session is already succeeded/failed, ignore duplicates safely.
  - Log repeated webhook events.

========================================================
9) CONCURRENCY / DOUBLE-SELL PROTECTION (Prod-leaning)
========================================================
- Implement reservation as an atomic DB operation using Prisma transaction:
  - When creating an order, only reserve if Product.status is active.
  - If not active, return an error "Product not available".
- On payment failure/cancel, release reservation to active.
- Consider setting Product.status to sold automatically when Order.status becomes completed (optional).
